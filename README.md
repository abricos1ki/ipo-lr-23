## Лабораторная работа №17. Работа с Django ORM

Разработаны модели данных интернет-магазина, настроена админ-панель,
реализованы ORM-запросы (фильтрация, сортировка, поиск, пагинация),
шаблоны для отображения списка товаров и страницы детализации.

### Модели (shop/models.py)

- **Category** ("Категория товара") — `name`, `description`.
- **Manufacturer** ("Производитель") — `name`, `country`, `description`.
- **Product** ("Товар") — `name`, `description`, `photo`, `price`,
  `stock_quantity`, `category` (FK), `manufacturer` (FK).
  Цена и количество на складе не могут быть отрицательными
  (`MinValueValidator` + проверка в `clean()`).
- **Cart** ("Корзина") — `user` (OneToOne к `User`), `created_at`.
  Метод `total_cost()` (он же `общая_стоимость`) считает сумму по всем
  элементам корзины.
- **CartItem** ("Элемент корзины") — `cart` (FK), `product` (FK),
  `quantity`. Метод `item_cost()` (он же `стоимость_элемента`) возвращает
  `товар.цена * количество`. В `clean()` проверяется, что количество не
  превышает остаток на складе.

Все связи `ForeignKey`/`OneToOneField` используют `on_delete=models.CASCADE`.

### Админ-панель

Все 5 моделей зарегистрированы в `shop/admin.py` со списками,
фильтрами и поиском. Элементы корзины отображаются inline внутри
карточки корзины.

### Страницы каталога

| URL                       | Описание                                                       |
|-----------------------------|------------------------------------------------------------------|
| `/products/`                | Список товаров: фильтр по категории/производителю, поиск (`q`), сортировка (`sort`), пагинация по 10 товаров на странице |
| `/products/<id>/`           | Страница детализации товара (404, если товар не найден)         |

GET-параметры `/products/`:
- `category=<id>` — фильтр по категории
- `manufacturer=<id>` — фильтр по производителю
- `q=<текст>` — поиск по названию или описанию (через `Q`-объекты)
- `sort=price_asc | price_desc | name` — сортировка
- `page=<номер>` — пагинация

## Лабораторная работа №18. Реализация CRUD-операций с Django ORM

Реализован полный цикл работы с корзиной пользователя (CRUD), каталог
товаров под `/catalog/` и аутентификация с ограничением доступа к
корзине через `@login_required`.

### URL-маршруты

| URL                              | Метод | Описание                                              |
|------------------------------------|-------|--------------------------------------------------------|
| `/catalog/`                        | GET   | Список товаров (фильтры, поиск, сортировка, пагинация) |
| `/catalog/<int:pk>/`               | GET   | Детальная информация о товаре                          |
| `/cart/`                           | GET   | Просмотр корзины (только для аутентифицированных)       |
| `/cart/add/<int:product_id>/`      | POST  | Добавление товара в корзину                             |
| `/cart/update/<int:item_id>/`      | POST  | Обновление количества товара в корзине                  |
| `/cart/remove/<int:item_id>/`      | POST  | Удаление товара из корзины                              |
| `/accounts/login/`                 | GET/POST | Вход в систему (стандартные auth-views Django)       |
| `/accounts/logout/`                | POST  | Выход из системы                                        |

(Старые маршруты `/products/` и `/products/<id>/` из лабораторной №17
оставлены для обратной совместимости и указывают на те же
представления, что и `/catalog/`.)

### Представления (shop/views.py)

- **`product_list`** — список товаров с фильтрацией по категории/производителю и поиском через `Q`-объекты (используется и для `/catalog/`, и для `/products/`).
- **`product_detail`** — детальная страница товара, `get_object_or_404` обрабатывает случай "товар не найден" (404).
- **`add_to_cart`** — добавление товара в корзину (`@login_required`); если товар уже в корзине — количество увеличивается; проверяется, что итоговое количество не превышает остаток на складе.
- **`update_cart`** — обновление количества (`@login_required`); валидация на остаток на складе и на положительное значение.
- **`remove_from_cart`** — удаление элемента корзины (`@login_required`), доступ только к своим элементам (`cart__user=request.user`).
- **`cart_view`** — отображение всех элементов корзины с общей стоимостью (`@login_required`).

### Аутентификация

- Используются стандартные `django.contrib.auth.urls` (`/accounts/login/`, `/accounts/logout/`).
- `LOGIN_URL`, `LOGIN_REDIRECT_URL`, `LOGOUT_REDIRECT_URL` настроены в `settings.py`.
- Шаблон входа: `shop/templates/registration/login.html`.
- Доступ к `/cart/` и операциям над корзиной ограничен декоратором `@login_required` — неаутентифицированный пользователь перенаправляется на страницу входа с параметром `?next=`.
- Тестовые пользователи (создаются командой `seed_data`): `anna`, `ivan`, `olga`, `dmitry`, `maria`, пароль `password123`.

### Шаблоны (добавлены/обновлены)

- `shop/templates/shop/product_list.html` — добавлена форма "Добавить в корзину" для каждого товара (количество + кнопка), скрыта для неаутентифицированных пользователей.
- `shop/templates/shop/product_detail.html` — аналогичная форма добавления в корзину.
- `shop/templates/shop/cart.html` — таблица с товарами в корзине: название, цена за штуку, форма обновления количества, стоимость элемента, форма удаления, итоговая сумма.
- `shop/templates/shop/base.html` — добавлены ссылка на корзину, имя пользователя, форма выхода, блок вывода сообщений (`django.contrib.messages`) об успехе/ошибке операций.
- `shop/templates/registration/login.html` — форма входа.

## Лабораторная работа №19. Работа с почтовым сервером и сторонними библиотеками

Реализовано оформление заказа: создание заказа на основе корзины,
генерация чека в формате Excel (`openpyxl`) и отправка чека на email
покупателя через `django.core.mail`.

### URL-маршрут

| URL            | Метод     | Описание                                          |
|------------------|-----------|------------------------------------------------------|
| `/checkout/`     | GET/POST  | Страница оформления заказа (только для аутентифицированных) |

### Представление `checkout` (shop/views.py)

- `@login_required` — доступно только аутентифицированным пользователям.
- **GET** — показывает форму с содержимым корзины, полем адреса доставки и комментарием.
- **POST**:
  1. Проверяет, что корзина не пуста, адрес доставки указан, а у пользователя есть email.
  2. Дополнительно проверяет остатки на складе для всех позиций корзины.
  3. В одной транзакции (`transaction.atomic`) создаёт `Order` и связанные `OrderItem` (с фиксацией цены на момент покупки), списывает товары со склада и очищает корзину.
  4. Генерирует чек в формате `.xlsx` (модуль `shop/receipts.py`, `openpyxl`).
  5. Отправляет письмо с вложенным чеком на email пользователя через `EmailMessage`.
  6. При ошибке (например, сбой SMTP) показывает сообщение об ошибке и не теряет данные заказа.

### Модели (добавлены в shop/models.py)

- **Order** ("Заказ") — `user` (FK), `delivery_address`, `comment`, `created_at`. Метод `total_cost()` считает сумму по позициям заказа.
- **OrderItem** ("Позиция заказа") — `order` (FK), `product` (FK), `quantity`, `price` (цена товара на момент заказа — фиксируется, чтобы не зависеть от последующих изменений цены товара).

### Генерация чека (shop/receipts.py)

Функция `build_order_receipt(order)` строит Excel-книгу с помощью `openpyxl`:
шапка с номером заказа, датой, покупателем и адресом доставки, таблица
позиций (товар / цена / количество / сумма) и итоговая строка. Возвращает
содержимое в виде `BytesIO`, готовое для вложения в письмо.

### Настройка email (settings.py)

Параметры почты задаются через переменные окружения `EMAIL_HOST_USER` и
`EMAIL_HOST_PASSWORD` (см. `.env.example`):

- Если переменные заданы — используется реальный SMTP-сервер Yandex (`smtp.yandex.ru`, порт 465, SSL).
- Если не заданы — используется `django.core.mail.backends.console.EmailBackend`,
  и письма с чеком выводятся в консоль сервера разработки. Это позволяет
  проверить весь функционал без реального почтового аккаунта.

Для Yandex-почты в `EMAIL_HOST_PASSWORD` нужно использовать **пароль
приложения** (создаётся в настройках Яндекс ID), а не обычный пароль
от почты.

### Шаблон (shop/templates/shop/checkout.html)

Показывает таблицу товаров из корзины с итоговой суммой и форму с полями
"Адрес доставки" (обязательное) и "Комментарий" (необязательное), кнопка
"Подтвердить заказ". Ссылка на оформление заказа добавлена на странице
корзины (`cart.html`).

## Лабораторная работа №20. Реализация REST API с Django REST Framework

Реализован REST API для всех моделей магазина с использованием
Django REST Framework: сериализаторы, `ModelViewSet`, маршрутизация
через `DefaultRouter`, доступ только для аутентифицированных пользователей.

### Установка и настройка DRF

- `rest_framework` подключён в `INSTALLED_APPS`.
- В `settings.py` добавлен блок `REST_FRAMEWORK`:
  - `DEFAULT_AUTHENTICATION_CLASSES`: `SessionAuthentication` (для входа через браузерный интерфейс DRF) и `BasicAuthentication` (для запросов из Postman/curl с логином и паролем).
  - `DEFAULT_PERMISSION_CLASSES`: `IsAuthenticated` -- доступ к API только для вошедших пользователей.
  - Постраничная разбивка результатов (`PageNumberPagination`, 10 объектов на страницу).

### Сериализаторы (shop/serializers.py)

Созданы сериализаторы (`ModelSerializer`) для всех моделей магазина:

- `CategorySerializer`, `ManufacturerSerializer` -- простые сериализаторы.
- `ProductSerializer` -- добавлены read-only поля `category_name`/`manufacturer_name` для удобства чтения, валидация цены и количества на складе (не могут быть отрицательными).
- `CartItemSerializer`, `OrderItemSerializer` -- добавлено вычисляемое поле `item_cost` (цена × количество) и `product_name`.
- `CartSerializer`, `OrderSerializer` -- включают вложенный список элементов (`items`, только для чтения) и `total_cost` (общая стоимость).
- В `CartItemSerializer` реализована валидация: количество не может превышать остаток товара на складе.

### API-представления (shop/api_views.py)

Все представления реализованы через `viewsets.ModelViewSet`, что
автоматически даёт полный набор CRUD-операций (list, create, retrieve,
update, partial_update, destroy):

- `CategoryViewSet`, `ManufacturerViewSet` -- доступны всем аутентифицированным пользователям.
- `ProductViewSet` -- поддерживает фильтрацию по `?category=<id>` и `?manufacturer=<id>`.
- `CartViewSet`, `CartItemViewSet`, `OrderViewSet`, `OrderItemViewSet` -- переопределён `get_queryset()`, чтобы обычный пользователь видел и мог изменять только свои собственные корзину/заказы (суперпользователь видит все). Попытка обратиться к чужому объекту по id возвращает `404`.

### URL-маршруты (shop/api_urls.py)

Используется `DefaultRouter` из DRF, который автоматически генерирует
маршруты для каждого зарегистрированного `ViewSet`:

| Эндпоинт                  | Методы                                  |
|------------------------------|--------------------------------------------|
| `/api/categories/`           | GET, POST, GET/PUT/PATCH/DELETE `<id>/`     |
| `/api/manufacturers/`        | GET, POST, GET/PUT/PATCH/DELETE `<id>/`     |
| `/api/products/`             | GET, POST, GET/PUT/PATCH/DELETE `<id>/` (+ фильтры `?category=`, `?manufacturer=`) |
| `/api/carts/`                | GET, POST, GET/PUT/PATCH/DELETE `<id>/` (только свои данные) |
| `/api/cart-items/`           | GET, POST, GET/PUT/PATCH/DELETE `<id>/` (только свои данные) |
| `/api/orders/`               | GET, POST, GET/PUT/PATCH/DELETE `<id>/` (только свои данные) |
| `/api/order-items/`          | GET, POST, GET/PUT/PATCH/DELETE `<id>/` (только свои данные) |
| `/api-auth/login/`           | Вход в браузерный интерфейс DRF              |

API подключён в главном `yoga_shop/urls.py` под префиксом `/api/`.

### Аутентификация и тестирование

Доступ к API возможен:

1. **Через сессию** -- войти на сайте обычным способом (`/accounts/login/`) и обращаться к `/api/...` из того же браузера (или передавая cookies в Postman/curl).
2. **Через Basic Auth** -- например, `curl -u anna:password123 http://127.0.0.1:8000/api/products/`.
3. **Через браузерный интерфейс DRF** -- открыть `/api/products/` в браузере, появится удобный HTML-интерфейс с формами для POST/PUT и кнопкой входа (`/api-auth/login/`).

Без аутентификации API возвращает `403 Forbidden`.

Пример проверки через curl (полный CRUD для товара):

```bash
# Список товаров
curl -u anna:password123 http://127.0.0.1:8000/api/products/

# Создание товара
curl -u anna:password123 -X POST http://127.0.0.1:8000/api/products/ \
  -H "Content-Type: application/json" \
  -d '{"name": "Новый товар", "description": "Описание", "price": "100.00", "stock_quantity": 5, "category": 1, "manufacturer": 1}'

# Обновление товара (PUT, id=35)
curl -u anna:password123 -X PUT http://127.0.0.1:8000/api/products/35/ \
  -H "Content-Type: application/json" \
  -d '{"name": "Обновлённый товар", "description": "Описание", "price": "120.00", "stock_quantity": 3, "category": 1, "manufacturer": 1}'

# Удаление товара
curl -u anna:password123 -X DELETE http://127.0.0.1:8000/api/products/35/
```

## Установка и запуск

1. Клонировать репозиторий:
   ```bash
   git clone <ссылка на репозиторий>
   cd <папка проекта>
   ```

2. Создать и активировать виртуальное окружение:

   **Windows:**
   ```bash
   python -m venv venv
   venv\Scripts\activate
   ```

   **Linux / macOS:**
   ```bash
   python3 -m venv venv
   source venv/bin/activate
   ```

3. Установить зависимости:
   ```bash
   pip install -r requirements.txt
   ```

4. Применить миграции:
   ```bash
   python manage.py migrate
   ```

5. (Опционально) Заполнить базу тестовыми данными:
   ```bash
   python manage.py seed_data
   ```

6. (Опционально) Создать суперпользователя для входа в админ-панель:
   ```bash
   python manage.py createsuperuser
   ```

7. (Опционально) Настроить реальную отправку email через Yandex SMTP:
   ```bash
   export EMAIL_HOST_USER=your_login@yandex.ru
   export EMAIL_HOST_PASSWORD=your_app_password
   ```
   (см. `.env.example`). Без этих переменных чек будет выводиться в
   консоль сервера разработки вместо реальной отправки -- этого
   достаточно, чтобы проверить весь функционал локально.

8. Запустить сервер разработки:
   ```bash
   python manage.py runserver
   ```

9. Открыть в браузере:
   - Главная страница: [http://127.0.0.1:8000/](http://127.0.0.1:8000/)
   - Каталог товаров: [http://127.0.0.1:8000/catalog/](http://127.0.0.1:8000/catalog/)
   - Корзина (требует входа): [http://127.0.0.1:8000/cart/](http://127.0.0.1:8000/cart/)
   - Оформление заказа (требует входа): [http://127.0.0.1:8000/checkout/](http://127.0.0.1:8000/checkout/)
   - Вход в систему: [http://127.0.0.1:8000/accounts/login/](http://127.0.0.1:8000/accounts/login/)
   - Админ-панель: [http://127.0.0.1:8000/admin/](http://127.0.0.1:8000/admin/)
   - REST API (требует входа или Basic Auth): [http://127.0.0.1:8000/api/products/](http://127.0.0.1:8000/api/products/)
   - Вход в браузерный интерфейс API: [http://127.0.0.1:8000/api-auth/login/](http://127.0.0.1:8000/api-auth/login/)
